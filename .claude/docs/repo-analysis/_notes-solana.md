# Solana-ecosystem notes (cross-repo)

- **Whirlpool decoding via SDK is ~80% free, ~20% on us.** Free: Position account decode, owner→positions discovery (including Token-2022 and bundles), fee/reward quote math, tick/price/liquidity math, all instruction codecs. On us: classifying each instruction into tax-event categories (entry/add/remove/harvest/compound/close/swap), pairing instructions with their sibling SPL-Token transfers to extract amounts moved, USD pricing per timestamp, dedup, persistence.
- **Three npm packages to pick from.** `@orca-so/whirlpools-sdk` (legacy, web3.js, most docs target it), `@orca-so/whirlpools` + `@orca-so/whirlpools-client` + `@orca-so/whirlpools-core` (new, `@solana/kit`-based, WASM math). For a 2026 greenfield CLI: go with the new stack; but keep `whirlpools-sdk` around since several example snippets and `collectFeesQuote` are only pure-TS there.
- **IDL source:** checked-in under `target/idl/whirlpool.json` after `anchor build`; distributed inside `@orca-so/whirlpools-sdk/dist/artifacts/whirlpool.json`; also on jsdelivr. Anchor 0.32 format, compatible with `@debridge-finance/solana-transaction-parser`.
- **Yes, tx-parser + Orca IDL replaces Helius Enhanced Transactions for Whirlpool.** The parser decodes every Whirlpool instruction name, args, and account labels. Inner instructions (CPIs) are handled via `flattenTransactionResponse`, which is critical because Whirlpool is typically invoked via Jupiter routes or custom routers — the top-level ix is Jupiter, and Whirlpool swap/collectFees/decreaseLiquidity appears only in `meta.innerInstructions`. The flattener produces one ordered list interleaving outer + CPI ixs so downstream classification is a single pass.
- **CPI handling strategy:** always call `flattenTransactionResponse` first, then parse. Keep the original `callIndex` as our per-ix identity so we can cross-ref `parseLogs` output for error diagnostics. Track `depth` (outer=0, CPI=1, nested CPI=2) on each ix — useful to decide whether a `collectFees` is user-initiated (outer) or part of a larger compound (CPI inside `increaseLiquidity`).
- **`update_fees_and_rewards` gotcha is not a blocker for historical decoding.** It matters only for "quote NOW without signing a tx." For parsing past transactions, the actual fee amounts are the SPL token-transfer `amount` fields in the CPI inner instructions accompanying `collectFees`/`collectFeesV2` — we read those directly.
- **Token-2022 + transfer fees** — Whirlpool V2 instructions route through Token-2022 for tokens with transfer hooks/fees. The fee-quote helpers already subtract transfer fees (see `transferFee` field in `CollectFeesQuote`). We need to mirror that when computing realized proceeds.
- **Position NFT discovery:** `fetchPositionsForOwner` in the new SDK walks both TOKEN_PROGRAM and TOKEN_2022_PROGRAM token accounts for the owner and filters to 1-supply mints that decode as a Position or PositionBundle PDA. This is the one-call path to "what are all of Felix's current Whirlpool positions."
- **solana-etl is the wrong tool.** Block-level ETL, no wallet filter. Ignore unless we later want bulk analytics; then prefer the BigQuery public dataset it feeds rather than running it ourselves.
- **Dep alignment hazard:** tx-parser depends on `@coral-xyz/anchor ^0.31.1`, Whirlpool is built with Anchor 0.32. Minor-version IDL parsing normally works but worth a smoke test on a real tx early. If it breaks, `src/legacy.idl.converter.ts` in tx-parser handles older formats and we could write the reverse if needed.
