# whirlpools

**Location:** ~/Code/Misc/onchain/whirlpools
**Language:** Rust (on-chain program + rust-sdk) + TypeScript (ts-sdk + legacy-sdk)
**License:** Orca / proprietary (see `LICENSE` in repo — "SEE LICENSE IN LICENSE" per package.json; source-available)
**Maintenance:** Very active. Official Orca monorepo. Audits through Aug 2025. Packages published to npm under `@orca-so/*`.

## Purpose
Orca Whirlpools is Solana's concentrated-liquidity AMM (CLMM). This monorepo contains:
- The on-chain Anchor program (`programs/whirlpool/`) at program ID `whirLbMiicVdio4qvUfM5KAg6Ct8VwpYzGff3uctyCc`.
- Multiple TS SDKs for interacting with it (new `@solana/kit`-based and legacy `@solana/web3.js`-based).
- A Rust SDK mirror.
- The Anchor IDL generated from the program.

## Architecture
Nx-managed monorepo.

- `programs/whirlpool/` — the on-chain Rust/Anchor program. IDL is generated from here into `target/idl/whirlpool.json` by `anchor build`.
- `ts-sdk/client/` — package `@orca-so/whirlpools-client` v6.2.1. Codama-generated low-level client: account decoders, instruction builders, PDAs, discriminators. Does NOT include math / quote helpers.
- `ts-sdk/core/` — package `@orca-so/whirlpools-core` v3.1.0. Shared math (tick math, fee/reward quoting, liquidity <-> amounts, price math). Actually Rust compiled to WASM (`wasm-pack`) + JS bindings.
- `ts-sdk/whirlpool/` — package `@orca-so/whirlpools` v7.0.2. High-level action helpers: `harvestPositionInstructions`, `collectFeesQuote`, `fetchPositionsForOwner`, `decreaseLiquidity`, `increaseLiquidity`, etc. Uses `@solana/kit` (v5), not web3.js.
- `legacy-sdk/whirlpool/` — package `@orca-so/whirlpools-sdk` v0.20.0. The web3.js/Anchor-0.32 generation. Still maintained; includes hand-written TS math under `src/utils/math/` and explicit `collectFeesQuote` / `collectRewardsQuote` under `src/quotes/public/`. **This is the SDK most existing examples and tutorials target.**
- `rust-sdk/` — Rust mirror.
- `idl/README.md` — points to `https://cdn.jsdelivr.net/npm/@orca-so/whirlpools-sdk@latest/dist/artifacts/whirlpool.json` as the distributed IDL. The IDL is also produced at `target/idl/whirlpool.json` after `anchor build` and is consumed by `ts-sdk/client/codama.js` to codegen the Codama client.

## Concrete value for our decoder

- **TS SDK as npm dep: yes, directly.** Two options:
  - `@orca-so/whirlpools-sdk` (legacy, web3.js-based) — closest to the "classic" examples. Has `PoolUtil`, `PositionUtil`, `PriceMath`, `TickUtil`, and the `collectFeesQuote` / `collectRewardsQuote` functions we need for position valuation.
  - `@orca-so/whirlpools` + `@orca-so/whirlpools-client` + `@orca-so/whirlpools-core` (new, `@solana/kit`-based). More modular; WASM math.
  - For a new TypeScript CLI starting in 2026, the new `@orca-so/whirlpools` is the forward-looking choice. But if we want simple decoding (not transaction building), `@orca-so/whirlpools-client` alone is enough — Codama-generated, includes Position account decoder with the `POSITION_DISCRIMINATOR` and all instruction codecs.
- **Position-decoding / valuation functions we get for free:**
  - `fetchPositionsForOwner(rpc, owner)` — scans wallet's token accounts, filters to Whirlpool position NFTs (including bundles and Token-2022), fetches and decodes Position accounts. File: `ts-sdk/whirlpool/src/position.ts`.
  - `decodePosition` from `@orca-so/whirlpools-client` — Borsh/Codama decode of a raw Position account.
  - `collectFeesQuote(whirlpool, position, tickLower, tickUpper, tokenExtCtx)` — exact unharvested fee calculation using fee-growth checkpoints. Returns `{feeOwedA, feeOwedB, transferFee}`. File: `legacy-sdk/whirlpool/src/quotes/public/collect-fees-quote.ts`.
  - `collectRewardsQuote(...)` — same for up to 3 reward streams. File: `.../collect-rewards-quote.ts`.
  - `getTokenAmountsFromLiquidity` / `PositionUtil.getTokenAmountsFromLiquidity` — turns `liquidity + tickLower + tickUpper + currentSqrtPrice` into underlying token amounts. This is what we need for "current underlying value of the LP position."
  - `PriceMath.sqrtPriceX64ToPrice`, `TickUtil.sqrtPriceX64ToTickIndex` — price <-> tick conversion.
- **IDL JSON:** not shipped as a standalone checked-in file in the repo root. It's generated by `anchor build` into `target/idl/whirlpool.json`, and bundled by `@orca-so/whirlpools-sdk` at `dist/artifacts/whirlpool.json` (downloadable from the jsdelivr URL above). **For our decoder: just `import whirlpoolIdl from '@orca-so/whirlpools-sdk/dist/artifacts/whirlpool.json'` at runtime, or fetch it once and check it in.** That JSON is Anchor 0.32 format, which is exactly what `@debridge-finance/solana-transaction-parser` expects.
- **`update_fees_and_rewards` gotcha:** confirmed relevant. `legacy-sdk/whirlpool/src/instructions/update-fees-and-rewards-ix.ts` contains the instruction. Comment notes `LiquidityZero` error if already updated. `collectFeesQuote` reads `feeGrowthCheckpointA/B` from the Position account — these checkpoints are only advanced when `update_fees_and_rewards` (or increaseLiquidity/decreaseLiquidity/collectFees) runs on-chain. For offline quoting *from current state* we re-do the math ourselves (the quote function does `feeGrowthInside - checkpoint`), so we don't need to send the ix. But: if we're reconstructing historical harvest amounts from a transaction, the actual fee transfer is inside the `collectFees` / `collectFeesV2` CPI, so we just read the SPL token transfer amounts — we don't need to replay the fee math. The `update_fees_and_rewards` gotcha mainly matters for "quote NOW, no state change" flows; not a problem for tax-event decoding.
- **Position NFT → underlying amounts:** requires tick math, yes, but `@orca-so/whirlpools-core` (WASM) and `legacy-sdk/.../utils/math/` ship that math. We do NOT roll our own.

## Key files

- `ts-sdk/whirlpool/src/position.ts` — owner-to-positions discovery, bundle handling.
- `ts-sdk/whirlpool/src/harvest.ts` — ready-made harvest flow + fee/reward quote composition.
- `ts-sdk/client/src/generated/accounts/position.ts` — `Position` account layout + `POSITION_DISCRIMINATOR`.
- `ts-sdk/client/src/generated/instructions/` — codecs for every Whirlpool instruction (open/close position, increase/decrease liquidity, collectFees, collectFeesV2, collectReward, swap, etc.).
- `legacy-sdk/whirlpool/src/quotes/public/collect-fees-quote.ts` and `collect-rewards-quote.ts` — the canonical fee/reward math. Readable reference even if we use the new SDK.
- `legacy-sdk/whirlpool/src/utils/math/` — tick/price/token-amount math in pure TS.
- `idl/README.md` — how to grab the IDL JSON.

## Integration sketch

For our tax decoder: take Helius (or a plain RPC) `getSignaturesForAddress` + `getTransaction` stream for the user's wallet, hand each tx to a `SolanaParser` (from solana-tx-parser-public) initialized with the Whirlpool IDL JSON pulled from `@orca-so/whirlpools-sdk/dist/artifacts/whirlpool.json`. For each flattened instruction whose programId is `whirLbMi...`, dispatch on the IDL instruction name (`openPosition`, `openPositionWithTokenExtensions`, `increaseLiquidity`, `increaseLiquidityV2`, `decreaseLiquidity`, `decreaseLiquidityV2`, `collectFees`, `collectFeesV2`, `collectReward`, `collectRewardV2`, `closePosition`, `swap`, `swapV2`, `twoHopSwap*`). Pair each with its sibling SPL `transfer`/`transferChecked` inner instructions (same flattened-tx shape) to get actual token amounts moved. For "snapshot current position value" at report time, use `fetchPositionsForOwner` + `collectFeesQuote` + `getTokenAmountsFromLiquidity` from the SDK. That's the whole path — decode historical events from instruction data+transfers, snapshot current unrealized state from SDK calls.

## Gaps

- Our decoder still has to classify each instruction into tax-event categories (entry vs add vs remove vs fee-harvest vs swap vs close), attribute amounts, link to USD prices, dedupe across retries, and persist to SQLite. The SDK does none of that.
- Mapping `collectFees` token transfer amounts back to "fees" vs. `decreaseLiquidity` principal withdrawal requires instruction-level discrimination — the SDK won't pre-sort them for us.
- Multi-tx flows (e.g. compounding = collectFees then increaseLiquidity) need our own grouping logic.
- USD pricing at instruction timestamp is entirely on us.
- The new `@orca-so/whirlpools` requires `@solana/kit@^5` (peer dep). Mixing with older `@solana/web3.js` tooling is awkward — pick one lane.
